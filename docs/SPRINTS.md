# План спринтов — лендинг по продаже бензина

**Референс:** [TZ.md](./TZ.md)  
**Срок проекта:** до 10 дней  
**Текущий статус:** см. [STATUS.md](./STATUS.md)  
**Контекст для новой сессии:** см. [CONTEXT.md](./CONTEXT.md)

---

## Принципы разбивки

1. Каждый спринт — законченный инкремент с видимым результатом.
2. В конце спринта обновляются `STATUS.md` и `CONTEXT.md`.
3. Создаётся чекпоинт `docs/sprints/sprint-NN-checkpoint.md`.
4. Новая сессия начинается с чтения `CONTEXT.md` → `STATUS.md` → чекпоинта последнего спринта.

---

## Спринт 0 — Подготовка и архитектура

**Цель:** зафиксировать стек, структуру проекта и модель данных для перезапусков.

### Задачи

- [ ] Выбрать CMS/стек (критерии: админка без разработчика, лёгкое копирование, скорость).
- [ ] Инициализировать репозиторий и структуру каталогов.
- [ ] Спроектировать `config/site.yaml` — единый файл настроек:
  - компания (название, логотип, ИНН, ОГРН, адрес);
  - контакты (телефон, email, WhatsApp, Telegram, режим работы);
  - аналитика (ID Яндекс.Метрики, цели);
  - интеграции (Telegram bot token, chat_id, SMTP).
- [ ] Спроектировать `config/content.yaml` — редактируемые тексты всех 7 блоков.
- [ ] Зафиксировать дизайн-токены (цвета, шрифты, отступы) по ТЗ.
- [ ] Подобрать/заглушить изображения (бензовоз, топливо, доставка).
- [ ] Записать решения в `DECISIONS.md`.

### Критерии готовности

- Есть рабочий каркас проекта (запускается локально).
- Конфиг отделён от вёрстки — смена бренда = правка YAML, не кода.
- Документирован процесс копирования на новый домен.

### Артефакты контекста

- `docs/DECISIONS.md`
- `docs/sprints/sprint-00-checkpoint.md`
- обновлённые `STATUS.md`, `CONTEXT.md`

**Оценка:** 1 день

## Спринт 1 — Каркас, шапка, главный экран

**Цель:** первый рабочий экран с навигацией и адаптивной вёрсткой.

### Задачи

- [ ] Базовый HTML-шаблон, CSS-переменные, сброс стилей.
- [ ] Шапка: логотип, телефон (кликабельный `tel:`), WhatsApp, Telegram.
- [ ] Меню: `Главная | Топливо | Преимущества | Доставка | Как заказать | Контакты`.
- [ ] Плавный скролл к якорям (`scroll-behavior` / JS).
- [ ] Блок Hero:
  - заголовок, подзаголовок, 3 преимущества;
  - кнопка «Рассчитать стоимость»;
  - фоновое изображение бензовоза;
  - заглушка модальной формы (без отправки).
- [ ] Адаптив: desktop / tablet / mobile.
- [ ] Мобильное бургер-меню.

### Критерии готовности

- Hero и шапка соответствуют текстам из TZ.md §1.
- Навигация работает на всех разрешениях.
- Контент подтягивается из `config/content.yaml`.

### Артефакты контекста

- `docs/sprints/sprint-01-checkpoint.md`
- скриншоты или описание проверенных брейкпоинтов
- обновлённые `STATUS.md`, `CONTEXT.md`

**Оценка:** 1–1.5 дня

## Спринт 4 — Формы заявок и UX

**Цель:** рабочие формы с валидацией, UTM и защитой от спама.

### Задачи

- [ ] Единый компонент формы (поля по TZ.md):
  - Имя, Телефон, Вид топлива, Объём, Адрес, Комментарий (в финальной).
- [ ] Модальная форма для кнопок Hero / карточек топлива / CTA.
- [ ] Встроенная форма в финальном блоке §7.
- [ ] Валидация полей (телефон, обязательные поля).
- [ ] Сообщение успеха: «Спасибо! Заявка успешно отправлена…».
- [ ] Захват UTM-меток из URL → скрытые поля / localStorage.
- [ ] Honeypot или простая captcha от спама.
- [ ] API-эндпоинт `POST /api/lead` (пока логирует в файл).

### Критерии готовности

- Все CTA-кнопки открывают/скроллят к рабочей форме.
- UTM передаются в payload заявки.
- Форма не отправляется при пустых обязательных полях.

### Артефакты контекста

- `docs/sprints/sprint-04-checkpoint.md`
- пример payload заявки в чекпоинте

## Спринт 5 — Интеграции: Telegram, Email

**Цель:** заявки доходят до менеджера.

### Задачи

- [ ] Отправка заявки в Telegram (Bot API → chat_id).
- [ ] Отправка заявки на email (SMTP / sendmail).
- [ ] Формат сообщения: имя, телефон, топливо, объём, адрес, UTM, время.
- [ ] Обработка ошибок и fallback (если Telegram недоступен — email всё равно уходит).
- [ ] Секреты вынесены в `.env` / `config/site.yaml` (не в git).
- [ ] `.env.example` с описанием переменных.
- [ ] Тестовая отправка задокументирована в чекпоинте.

### Критерии готовности

- Тестовая заявка приходит в Telegram и на почту.
- UTM отображаются в уведомлении.
- Инструкция по смене chat_id / email для перезапуска.

### Артефакты контекста

- `docs/sprints/sprint-05-checkpoint.md`
- `docs/INTEGRATIONS.md` — настройка Telegram и SMTP
- обновлённые `STATUS.md`, `CONTEXT.md`

**Оценка:** 1 день

## Спринт 6 — Аналитика, админка, деплой, перезапуск

**Цель:** production-ready сайт с метрикой и инструкцией переноса.

### Задачи

- [ ] Яндекс.Метрика + Вебвизор (ID из конфига).
- [ ] Цели Метрики:
  - отправка формы;
  - клик по телефону;
  - переход WhatsApp;
  - переход Telegram.
- [ ] Оптимизация скорости: сжатие изображений, минификация CSS/JS, lazy-load.
- [ ] Админ-панель или пошаговая инструкция редактирования `config/*.yaml`.
- [ ] `docs/DEPLOY.md` — деплой на хостинг.
- [ ] `docs/RELAUNCH.md` — копирование на новый домен за N шагов.
- [ ] Финальное QA: Chrome, Safari, мобильные, формы, метрика.
- [ ] Чеклист приёмки по TZ.md.

### Критерии готовности

- PageSpeed / Lighthouse — приемлемые показатели (LCP < 2.5s на desktop).
- Заказчик может сменить логотип, контакты, тексты без разработчика.
- Есть инструкция «скопировать сайт → сменить домен → сменить бренд».

### Артефакты контекста

- `docs/sprints/sprint-06-checkpoint.md`
- `docs/DEPLOY.md`, `docs/RELAUNCH.md`
- финальный `STATUS.md` (статус: **ГОТОВ**)
- финальный `CONTEXT.md`

**Оценка:** 1–1.5 дня

## Рекомендуемый стек (для спринта 0)

| Критерий | Рекомендация |
|----------|--------------|
| Перезапуск | Статика + YAML-конфиги (без привязки к БД) |
| Админка | Редактирование YAML через файловый менеджер хостинга или лёгкая веб-админка |
| Формы | PHP-endpoint или Node.js serverless function |
| Хостинг | Обычный shared/VPS: nginx + статика + один backend-скрипт |
| Альтернатива | Grav CMS — если нужна GUI-админка из коробки |

Окончательный выбор — в спринте 0, запись в `DECISIONS.md`.
